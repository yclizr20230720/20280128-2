# FAB IT Training Program - Capstone Project

## Overview

The capstone project is your opportunity to demonstrate everything you've learned over the 4-week training program. You'll present a complete analysis of the Equipment Performance Monitoring System, from requirements through project plan.

**Presentation Date**: End of Week 4 (Friday afternoon)  
**Duration**: 30 minutes presentation + 15 minutes Q&A  
**Audience**: Instructors, peers, senior engineers, management (optional)  
**Format**: Professional slide presentation with supporting documentation

---

## Project Scope

### Complete Equipment Performance Monitoring System Analysis

You will present your complete work from Weeks 2-4:

1. **Requirements Analysis** (Week 2 deliverable)
2. **System Architecture Design** (Week 3 deliverable)
3. **Project Management Plan** (Week 4 deliverable)

### Integration Requirement

Your capstone must demonstrate how these three components work together as a cohesive solution:
- Requirements drive the architecture design
- Architecture enables the requirements
- Project plan implements the architecture to meet requirements
- All decisions are traceable and justified

## Presentation Guidelines

### Content Guidelines

**Do:**
- Tell a story: Problem → Solution → Implementation
- Use visuals: Diagrams communicate better than text
- Highlight key decisions and trade-offs
- Show traceability: Requirements → Architecture → Plan
- Quantify benefits: Time savings, cost reduction, efficiency gains
- Be specific: Use real numbers, timelines, technologies
- Practice: Rehearse multiple times

**Don't:**
- Read slides verbatim
- Include too much text on slides
- Go into excessive technical detail
- Skip over important decisions
- Ignore FAB-specific considerations
- Exceed time limit

### Slide Design Guidelines

**Visual Design:**
- Use consistent template and colors
- One main idea per slide
- Limit text (6 lines max, 6 words per line)
- Use large, readable fonts (minimum 24pt)
- Include diagrams and visuals
- Use bullet points, not paragraphs

**Diagram Guidelines:**
- Use professional diagramming tools
- Ensure diagrams are readable (large enough)
- Use consistent notation
- Include legends if needed
- Highlight key elements
- Keep diagrams simple and focused

### Delivery Guidelines

**Preparation:**
- Practice presentation 3-5 times
- Time yourself
- Prepare for questions
- Test technology (laptop, projector, remote)
- Have backup (USB drive, cloud copy)

**During Presentation:**
- Arrive early, set up and test
- Start with confidence
- Make eye contact with audience
- Speak clearly and at moderate pace
- Use natural gestures
- Engage with audience
- Watch time
- Handle questions professionally

**Handling Questions:**
- Listen carefully to full question
- Pause before answering
- Answer concisely
- If you don't know, say so (and offer to follow up)
- Don't get defensive
- Thank questioner

**Slide 1: Title**
- Equipment Performance Monitoring System
- Your Name
- January 2025

**Slide 2: Executive Summary**
- Problem: Engineers spend 2-3 hours/day manually checking equipment, reactive maintenance causes unplanned downtime
- Solution: Real-time monitoring system with predictive maintenance
- Value: 80% reduction in checking time, 50% reduction in unplanned downtime, 3% increase in equipment availability

**Slide 3: Stakeholders and Needs**
- Process Engineers: Real-time visibility, trend analysis, mobile access
- Equipment Engineers: Predictive maintenance, health scoring, maintenance tracking
- Operators: Simple interface, clear alarm priorities, quick acknowledgment
- IT: Secure, scalable, maintainable, standards-compliant

**Slide 4: Key Requirements**
- Business Requirements: Reduce downtime, improve visibility, enable predictive maintenance
- Functional Requirements: Real-time status, parameter trends, alerts, health scoring
- Non-Functional Requirements: < 2 sec response, 99.9% availability, 150 equipment support

**Slide 5: User Stories (Top 3)**
- Story 1: Real-time equipment monitoring
- Story 2: Predictive maintenance alerts
- Story 3: Parameter trend analysis
(Include acceptance criteria for each)

**Slide 6: Process Improvement**
- As-Is: Manual checking, 2 hours/day, reactive
- To-Be: Automated monitoring, real-time alerts, proactive
- Quantified improvement: 80% time savings, 50% downtime reduction

**Slide 7: System Context**
- Context diagram showing system boundaries
- External interfaces: MES, Equipment (SECS/GEM), Users
- Data flows

**Slide 8: Component Architecture**
- Component diagram with layers
- Data Collection Layer: SECS/GEM interface, message queue
- Processing Layer: Business logic, alert engine, analytics
- Presentation Layer: Web UI, mobile app, API gateway
- Data Layer: Database, cache, time-series DB

**Slide 9: Technology Stack**
- Frontend: React (modern, component-based)
- Backend: Java Spring Boot (enterprise-grade, FAB standard)
- Database: PostgreSQL (transactional), InfluxDB (time-series)
- Message Broker: Apache Kafka (high throughput, durability)
- API: REST (simplicity, mobile support)

**Slide 10: Integration Architecture**
- Sequence diagram: Equipment → Data Collector → Kafka → Processing → Database → API → UI
- SECS/GEM integration details
- MES integration approach

**Slide 11: Architecture Decisions**
- Decision 1: Event-driven architecture (Why: Real-time, scalability, loose coupling)
- Decision 2: Microservices (Why: Independent scaling, technology flexibility)
- Decision 3: Time-series database (Why: Optimized for sensor data)

**Slide 12: Architecture Trade-offs**
- Microservices vs. Monolith: Chose microservices for scalability, accepting increased complexity
- SQL vs. NoSQL: Chose both (PostgreSQL for transactional, InfluxDB for time-series)
- REST vs. gRPC: Chose REST for simplicity and mobile support

**Slide 13: Project Timeline**

- Phase 2: Core Development (6 weeks)
- Phase 3: Integration & Testing (4 weeks)

**Slide 16: Top 5 Risks**
- Risk 1: Equipment integration issues (High probability, High impact)
  - Mitigation: Early prototyping, vendor support, dedicated integration engineer
- Risk 2: Production disruption (Medium probability, Critical impact)
  - Mitigation: Phased rollout, comprehensive testing, rollback plan
- Risk 3: Performance issues (Medium probability, High impact)
  - Mitigation: Load testing, performance monitoring, caching strategy
- Risk 4: Scope creep (High probability, Medium impact)
  - Mitigation: Change control process, prioritization framework
- Risk 5: Resource availability (Medium probability, Medium impact)
